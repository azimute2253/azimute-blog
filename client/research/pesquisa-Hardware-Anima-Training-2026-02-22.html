Part I — Understanding the Workloads

A. Inference Workload (Anima Runtime)

Características do Runtime
  • Múltiplos LLMs Simultâneos: Diferentes tamanhos para diferentes "layers" da consciência (1B-13B-70B rodando em paralelo)
  • Memory Core: Semantic search com embeddings (nomic-embed-text via Ollama), operações de vector database
  • Sub-Agents Paralelos: Múltiplas sessões executando simultaneamente, cada uma com seu próprio contexto
  • Real-Time Processing: Message routing, gating, tool execution, browser automation
  • Sleep Cycles: Batch jobs durante "noite" (consolidação de memória, relatórios, limpeza)
  • Infraestrutura: Web search, database ops, file I/O intensivo

Requisitos de VRAM (Inferência) [T]

Fonte: Modal.com ML Infrastructure Guide [T], WhiteFiber GPU Comparison 2025 [T]

B. Training Workload (Custom Neural Architecture)

Características do Treinamento
  • Novel Architecture: Não é fine-tuning LoRA — treinamento from scratch de arquitetura inspirada nos 9 Layers
  • Model Sizes: 7B (proof-of-concept) → 30B (intermediário) → 70B (produção)
  • Custom CUDA Kernels: Potencialmente necessários para implementar mecanismos únicos da arquitetura
  • Dataset: Corpus de pesquisa (~5MB) + curated training data (estimativa: 50-200GB tokenizados)
  • Experimentação: Muitos training runs para testar ideias arquiteturais (iteração rápida essencial)
  • Mixed Precision: FP16/BF16 training com gradient checkpointing para economizar VRAM

Requisitos de VRAM (Treinamento) [A] [T]

Method Precision 7B 13B 30B 70B
Full Training FP16 ~67 GB ~125 GB ~288 GB ~672 GB
LoRA Fine-Tuning FP16 ~15 GB ~28 GB ~63 GB ~146 GB
QLoRA (8-bit) INT8 ~9 GB ~17 GB ~38 GB ~88 GB
QLoRA (4-bit) INT4 ~5 GB ~9 GB ~20 GB ~46 GB

VRAM = Model Weights (FP16) + Optimizer States (AdamW: 3× FP32) + Gradients (FP16)

Para 7B: 14GB (weights) + 42GB (optimizer 8-bit) + 14GB (gradients) ≈ 70GB

Fonte: Modal.com "How much VRAM for fine-tuning" [A], Jarvislabs H100 Price Guide [A]

⚠️ Implicação Crítica: Custom architecture training from scratch requer VRAM total completo (~70GB para 7B, ~672GB para 70B). Fine-tuning (LoRA/QLoRA) reduziria drasticamente requisitos, mas não é aplicável para arquitetura proprietária inicial.

Part VI — Software Stack for Custom Architecture

Core Framework: PyTorch + Distributed Training

PyTorch 2.x (Primary DL Framework)

  • Why PyTorch: Dynamic computation graph → ideal para custom architecture experimentation
  • torch.compile: 2× speedup via graph optimization (TorchDynamo + TorchInductor)
  • FSDP (Fully Sharded Data Parallel): Train models >VRAM single-GPU via sharding
  • CUDA Support: Native CUDA kernel integration para mecanismos customizados

DeepSpeed (Microsoft Distributed Training)

  • ZeRO (Zero Redundancy Optimizer): Reduz memory footprint 4-8×
    • ZeRO-1: Shards optimizer states
    • ZeRO-2: Shards gradients também
    • ZeRO-3: Shards model parameters (treina 13B em 24GB VRAM)
  • Pipeline Parallelism: Divide model em stages, processa batches em pipeline
  • Gradient Checkpointing: Trade compute por memory (re-compute activations)
  • Mixed Precision (FP16/BF16): 2× faster training, 2× memory reduction

Megatron-LM (NVIDIA Large Model Training)

  • Tensor Parallelism: Split layers across GPUs (vs. FSDP model-parallel)
  • Sequence Parallelism: Split sequence dimension para long-context models
  • Expert Parallelism: Para Mixture-of-Experts architectures
  • Integration: DeepSpeed + Megatron-LM = Megatron-DeepSpeed (best of both)

Custom Kernel Development: CUDA vs Triton

Framework Complexity Performance Development Time Use Case
CUDA C++ High 100% (hand-optimized) Weeks-Months Performance-critical operations (attention, custom activations)
Triton (OpenAI) Medium 90-95% CUDA Days-Weeks Rapid prototyping de custom ops, fusion kernels
PyTorch Autograd Low 70-80% (overhead) Hours Prototyping inicial, baseline implementation

Recomendação: Start com PyTorch native ops → profile → identify bottlenecks → rewrite hot paths em Triton → se critical, otimizar em CUDA.

Inference Serving: vLLM + Ollama

vLLM (High-Throughput Inference Server)

  • PagedAttention: 20× throughput vs. HuggingFace Transformers
  • Continuous Batching: Dynamic batching sem padding waste
  • Quantization: INT8/INT4 on-the-fly (AWQ, GPTQ)
  • Throughput: H100 80GB → ~3,500-4,000 tokens/sec (Llama 70B)

Ollama (Local Model Management)

  • Model Registry: Download, manage, version local models
  • API Server: OpenAI-compatible REST API
  • Quantization: Automatic Q4/Q8 quantization para consumer GPUs
  • Use Case: Development/prototyping; production → vLLM

Brazil-Specific Costs: Import Taxes & Markup

⚠️ Importação de Hardware no Brasil

  • Imposto de Importação (II): 16% sobre CIF (Cost + Insurance + Freight)
  • IPI (Imposto sobre Produtos Industrializados): 0-15% dependendo do item
  • ICMS (Imposto sobre Circulação de Mercadorias): ~18% (varia por estado)
  • PIS/COFINS: ~11.75%
  • Total Effective Tax: ~60-80% sobre preço FOB (Free On Board)
  • Retail Markup Brasil: GPUs enterprise têm 40-60% markup adicional sobre preço US + taxes

Exemplo: H100 80GB SXM $30K USD → ~R$270K-330K no Brasil ($54K-66K) = 80-120% markup total

Fonte: Trade.gov Brazil Import Tariffs [A], LinkedIn Brazil AI Workstation Market [E], Reddit r/computadores hardware pricing [E]

Estratégias para Reduzir Custos no Brasil

  1. REDATA Program (Sep 2025): Incentivos fiscais para datacenters — isenção de II para equipamentos sem produção similar nacional. Aplicável a GPUs enterprise (A100/H100). [A]
  2. Compra via Empresa: CNPJ permite recuperação de ICMS/PIS/COFINS (reduz ~30% custo efetivo)
  3. Used/Refurb Market: A100 40GB usado $8K USD vs. $12K novo (33% desconto)
  4. Cloud Hybrid: Evita capex; Lambda/Jarvislabs cobram em USD direto (sem markup Brasil)
  5. Phased Investment: Começar Tier 1-2 local (menor impacto fiscal) + cloud burst para training
