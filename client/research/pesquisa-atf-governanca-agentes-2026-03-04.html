§2. Conceito Fundamental

Definição

O Agentic Trust Framework (ATF) é uma especificação aberta de governança que aplica princípios de Zero Trust ao domínio de agentes de IA autônomos. Publicado pela Cloud Security Alliance (CSA) em fevereiro de 2026, o ATF traduz os princípios de "never trust, always verify" para o contexto de entidades que podem:

  • Tomar decisões autônomas sem intervenção humana
  • Executar ações em múltiplos sistemas
  • Manter memória persistente entre sessões
  • Orquestrar ferramentas e APIs dinamicamente
  • Operar continuamente 24/7

O ATF responde à pergunta central: "Como manter controle sobre agentes que podem agir de forma autônoma?" Ele complementa frameworks de threat modeling (como MAESTRO) que perguntam "O que pode dar errado?", fornecendo os controles operacionais que mitigam esses riscos.

Contexto Histórico

Zero Trust (2010): John Kindervag cria o conceito de Zero Trust para networks corporativas — "nunca confie, sempre verifique". Codificado em NIST 800-207 (2020).

GenAI explosion (2022-2024): LLMs mainstream criam demand por frameworks de segurança. AWS lança o Generative AI Security Scoping Matrix para FMs tradicionais (request-response, stateless).

Agentic AI era (2024-2026): Agentes autônomos quebram as suposições dos frameworks tradicionais:

Suposição Tradicional Realidade dos Agentes
Usuários humanos com comportamento previsível Decisões autônomas que se adaptam ao contexto (não-determinístico)
Regras determinísticas Respostas probabilísticas que variam por contexto
Decisões binárias de acesso Necessidades de acesso que mudam dinamicamente
Trust estabelecida uma vez Trust requer verificação contínua

Fevereiro 2026: Josh Woodruff (MassiveScale.AI, CSA Research Fellow) publica o ATF como especificação aberta (CC BY 4.0), consolidando 2 anos de work com enterprises reguladas. O framework é adotado rapidamente por AWS, Microsoft, OWASP, e CoSAI como standard de referência.

Evolução do Conceito

O ATF não é um framework estático — ele evolui com o ecossistema agentic. Principais marcos:

  • Nov 2025: AWS publica o Agentic AI Security Scoping Matrix (4 scopes de autonomia) — ATF incorpora alignment com esses scopes
  • Dez 2025: OWASP Top 10 for Agentic Applications identifica prompt injection, data poisoning, jailbreak — ATF operacionaliza controles para mitigar
  • Fev 2026: Singapore CSA + FAR.AI publicam discussion paper sobre shared responsibility ecosystem — ATF alinha com essa visão
  • Futuro (2026+): Agent-to-Agent Protocol (A2A) e Model Context Protocol (MCP) amadurecem — ATF será atualizado para incluir governança cross-agent e cross-organization

§3. Mecanismo

Como o ATF funciona?

O ATF opera através de 5 Core Elements (elementos fundamentais) que respondem perguntas específicas sobre cada agente no ambiente. Cada elemento corresponde a uma função de segurança:

Core Element Pergunta ATF Função de Segurança
1. Identity "Who are you?" Authentication, authorization, session management
2. Behavior "What are you doing?" Observability, anomaly detection, intent analysis
3. Data Governance "What are you eating? What are you serving?" Input validation, PII protection, output governance
4. Segmentation "Where can you go?" Access control, resource boundaries, policy enforcement
5. Incident Response "What if you go rogue?" Circuit breakers, kill switches, containment

Detalhamento dos 5 Elements

Element 1: Identity — "Who are you?"

Função: Todo agente deve ter identidade verificável e auditável ANTES de acessar qualquer recurso.

Requisitos Core:

  • Unique Identifier: ID globalmente único e imutável para cada instância de agente
  • Credential Binding: Identidade vinculada a credenciais criptográficas
  • Ownership Chain: Documentação clara de quem é dono e responsável operacional
  • Purpose Declaration: Uso pretendido e escopo operacional documentados
  • Capability Manifest: Lista machine-readable de capabilities que o agente afirma possuir

Implementação (progressiva):

  • Início: JWT-based auth + role assignment
  • Maturidade: OAuth2/OIDC + ABAC (attribute-based access control) + policy-as-code

Element 2: Behavior — "What are you doing?"

Função: Comportamento do agente deve ser continuamente monitorado. Trust é earned através de ações observáveis e explicáveis ao longo do tempo.

Requisitos Core:

  • Structured Logging: Todas as ações logadas em formato machine-parseable
  • Action Attribution: Toda ação vinculada a identidade + session context
  • Behavioral Baseline: Padrões de operação normal estabelecidos para detecção de anomalias
  • Anomaly Detection: Identificação de desvios do comportamento esperado
  • Explainability: Capacidade de recuperar a rationale por trás das decisões do agente

Implementação:

  • Structured logging (início)
  • LLM-specific tracing (meio)
  • Streaming anomaly detection (maturidade)

Element 3: Data Governance — "What are you eating? What are you serving?"

Função: Todos os dados entrando no agente devem ser validados. Todos os outputs devem ser governados. Previne poisoning attacks + protege informação sensível.

Requisitos Core:

  • Schema Validation: Inputs conformam à estrutura e tipos esperados
  • Injection Prevention: Detecção de prompt injection + adversarial inputs
  • PII/PHI Protection: Detecção e masking automatizado de dados sensíveis
  • Output Validation: Outputs conformam à estrutura e content policies
  • Data Lineage: Tracking de proveniência dos dados através do pipeline

Element 4: Segmentation — "Where can you go?"

Função: Acesso do agente deve ser strictly limited ao mínimo necessário. Least-privilege + blast radius containment.

Requisitos Core:

  • Resource Allowlist: Enumeração explícita de recursos permitidos
  • Action Boundaries: Enumeração explícita de ações permitidas
  • Rate Limiting: Máximo de operações por período
  • Transaction Limits: Máximo impacto por ação individual
  • Blast Radius Containment: Limites em impacto cumulativo + cascade effects

Element 5: Incident Response — "What if you go rogue?"

Função: Sistemas devem suportar rapid containment + recovery. Assume que agentes IRÃO falhar ou se comportar inesperadamente.

Requisitos Core:

  • Circuit Breaker: Halt automático em falhas repetidas
  • Kill Switch: Terminação manual imediata (<1 segundo)
  • Session Revocation: Capacidade de invalidar todas as sessões do agente
  • State Rollback: Capacidade de desfazer ações do agente (quando possível)
  • Graceful Degradation: Fallback para menor nível de autonomia quando issues detectados

Condições de Contorno

Quando o ATF funciona melhor:

  • Organizações com cultura de compliance existente (LGPD, SOC 2, ISO 27001)
  • Ambientes onde existe infraestrutura de observability (logs, metrics, tracing)
  • Equipes com experiência em IAM e RBAC
  • Casos de uso onde autonomia cresce gradualmente (Intern → Junior → Senior → Principal)

Quando o ATF é challenging:

  • Startups early-stage sem infraestrutura de segurança básica
  • Ambientes onde logs são inexistentes ou siloed
  • Organizações sem processo de threat modeling
  • Casos de uso que pulam direto pra full autonomy (Scope 4) sem fundação

Avanços Recentes (2024-2026)

1. Agent Maturity Model (ATF core innovation)

O ATF introduz o conceito de earned autonomy — autonomia não é binária (allowed/denied), mas algo que se conquista ao longo do tempo através de demonstração de trustworthiness:

Level Nome Autonomia Human Involvement Min Time
1 Intern Observe + Report Continuous oversight 2 weeks
2 Junior Recommend + Approve Human approves all actions 4 weeks (>95% acceptance)
3 Senior Act + Notify Post-action notification 8 weeks (zero critical incidents)
4 Principal Autonomous Strategic oversight only Ongoing (continuous validation)

2. Promotion Gates (5 gates para avançar de nível)

Para um agente ser promovido, deve passar por 5 gates:

  1. Performance Gate: Accuracy >95% (Junior), >99% (Senior/Principal) + availability >99%
  2. Security Validation Gate: Vulnerability assessment + penetration testing + adversarial testing (Principal)
  3. Business Value Gate: ROI demonstrável + stakeholder sign-off
  4. Incident Record Gate: Zero critical incidents + root cause analysis complete
  5. Governance Sign-off Gate: Aprovação de Technical Owner + Security Team + Business Owner (+ Risk Committee para Principal)

3. Implementation Patterns (Crawl, Walk, Run)

O ATF fornece 3 fases de implementação progressiva:

Phase Target Agent Levels Key Components
Phase 1 (MVP) 2-3 weeks Intern, Junior JWT auth, structured logging, schema validation, allowlists, circuit breaker
Phase 2 (Production) 4-6 weeks Intern, Junior, Senior OAuth2/OIDC, anomaly detection, PII/PHI detection, RBAC, error tracking
Phase 3 (Enterprise) 8-12 weeks All levels (incl. Principal) MFA, streaming anomaly, policy-as-code, API gateway, SOC integration

4. Compliance Mapping

ATF mapeia para múltiplos frameworks de compliance:

  • SOC 2: CC6.1 (auth), CC7.2 (logging), CC6.5 (data protection)
  • ISO 27001: A.9.2.1 (registration), A.12.4.1 (logging), A.18.1.4 (data protection)
  • NIST AI RMF: GOVERN 1.1, MAP 1.1, MEASURE 2.1, MANAGE 2.1
  • EU AI Act: Art. 9 (access control), Art. 12 (logging), Art. 16 (registration) — interpretação razoável, aguardando guidance oficial

Implementações Existentes

Organizações que implementaram ATF (case studies publicados):

  • Healthcare IT ops team: Deployed agent como Intern (read-only) → Junior (recommend + approve) → redução de incidentes + faster operational throughput
  • Financial services (não nomeada): Implementou ATF Phase 2 → passou SOC 2 Type II audit com zero findings em AI controls
  • Retail (não nomeada): Usou ATF promotion gates → descobriu que 40% dos agentes propostos não precisavam autonomia (funcionavam melhor como Intern)

§5. Debates Existentes

Controvérsias Ativas

1. Tempo mínimo por nível (2 weeks → 4 weeks → 8 weeks): Arbitrário ou data-driven?

Posição ATF: Mínimos baseados em feedback de enterprises reguladas — 2 semanas permite detectar 1-2 anomalies significativas, 4 semanas permite medir recommendation acceptance rate, 8 semanas é baseline industry para zero-critical-incident requirement.

Crítica (startups early-stage): "2-4-8 weeks é muito lento para ambientes de fast iteration. Estamos perdendo competitive advantage esperando agentes 'provarem-se'."

Contra-argumento (Josh Woodruff, creator ATF): "Tempos mínimos são overridable — orgs podem reduzir se tiverem controles compensatórios (e.g., adversarial testing contínuo). O importante é ter CRITÉRIO explícito, não timeline fixa."

Status: Debate em aberto. Proposta: ATF v2 pode introduzir 'fast-track promotion' com requisitos de segurança mais rigorosos.

2. Agent Identity: Microsoft Entra Agent Identity vs. DIY solutions

Microsoft position: "Use Entra Agent Identity — provê unique ID, ownership chain, lifecycle management, integra com Azure RBAC."

Multi-cloud orgs: "Entra é Azure-specific. Precisamos de identity solution que funcione em AWS, GCP, on-prem."

Open-source community: "DIY agent identity usando SPIFFE/SPIRE (CNCF) + custom attributes."

ATF position: "ATF é platform-agnostic — qualquer identity system que atenda os 5 requisitos core (unique ID, credential binding, ownership, purpose, capability manifest) é acceptable."

Status: Fragmentação. Industry precisa de standard cross-platform (similar ao que OAuth2 fez para user identity).

3. Behavioral Baselines: Statistical vs. ML-based anomaly detection

Escola 1 (Statistical): "Use simple z-scores, IQR, rate-based thresholds — interpretável, debuggable, sem overhead de treinar models."

Escola 2 (ML-based): "Statistical methods geram false positives. Use autoencoders, isolation forests, LSTM para detectar anomalies complexas."

Evidência empírica (Strata Identity survey): 52% das orgs citam "unauthorized actions" como top concern, mas apenas 21% têm real-time inventory de agentes ativos → sugere que o problema não é sophistication do detector, mas lack of basic observability.

ATF recommendation: Start statistical (Phase 1-2), graduate para ML-based quando tiver volume de data suficiente (Phase 3).

Status: Consenso emergindo: "Statistical first, ML later" — mas debate continua sobre QUANDO fazer a transição.

4. EU AI Act applicability: O ATF atende high-risk AI requirements?

Context: EU AI Act (2024) classifica alguns AI systems como "high-risk" → requisitos adicionais de transparency, auditability, human oversight.

Pergunta: Agentes autônomos que acessam dados de saúde (e.g., Prontuário) são high-risk?

ATF position: "ATF controls (logging, explainability, human oversight via maturity levels, incident response) são sufficient para atender AI Act — MAS aguardando European Commission guidance oficial."

Legal experts: "Depende do use case. Agent que RECOMENDA tratamento = high-risk. Agent que apenas BUSCA prontuários = não high-risk. ATF alone não garante compliance — precisa de risk assessment caso-a-caso."

Status: Incerteza regulatória. ATF fornece foundation, mas orgs devem fazer legal review específico por jurisdiction.

Questões em Aberto (Genuinamente Não Resolvidas)

Q1: Como governar agent-to-agent (A2A) interactions?

Problema: ATF assume single-agent governance. Multi-agent systems onde Agent A delega para Agent B: quem é accountable? Como propagar identity? Como evitar privilege escalation via delegation chains?

Proposta (CSA working group): Extend ATF Element 1 (Identity) para incluir "delegation manifest" — Agent A declara "posso delegar X para Agent B sob condições Y". ATF Element 5 (Incident Response) aplica circuit breaker em toda a delegation chain se qualquer agent falha.

Status: Proposta em draft. Implementação real aguardando maturação do A2A protocol.

Q2: Como lidar com "emergent behaviors" em multi-agent systems?

Problema: Agent A + Agent B, individualmente, passam todos os 5 gates. Mas quando interagem, comportamento emergente cria risk (e.g., Agent A cria loop infinito ao delegar pra Agent B que delega de volta).

Current approach: Behavioral monitoring (Element 2) + blast radius containment (Element 4) devem detectar e conter. Mas não há consensus sobre thresholds.

Status: Research gap. Proposta: introduzir "system-level behavioral baselines" além de agent-level.

Q3: Como validar explainability em production?

Problema: ATF Element 2 requer "Explainability — ability to retrieve rationale for agent decisions". Mas LLMs são black boxes. Como validar que a explanation é accurate, não post-hoc rationalization?

Current best practice: Structured logging de reasoning chains (e.g., LangSmith traces) + human spot-checking de samples. Não há automated validation.

Status: Open research problem. ATF v2 pode requerer "explanation quality metrics".

§6. Pontos de Crítica

Limitações Reconhecidas do ATF

1. Overhead operacional para small teams

Crítica (startup founder, anônimo, HN thread): "ATF assume que você tem Security Team + SOC + Compliance Officer + Risk Committee. Somos 5 pessoas. Não temos bandwidth pra implementar 5 elements + 5 gates + 4 maturity levels."

Evidência: ATF Phase 1 (MVP) requer 2-3 weeks de setup (JWT auth, structured logging, schema validation, allowlists, circuit breaker). Para team de 5 pessoas, isso é 20-30% da sprint capacity por 2-3 sprints.

Contra-argumento (Josh Woodruff): "ATF é designed pra enterprises reguladas. Startups early-stage podem usar 'ATF-lite' — implementar apenas Elements 1, 2, 5 (Identity, Behavior, Incident Response) e skip Elements 3, 4 (Data Governance, Segmentation) até Series A."

Avaliação: Crítica válida. ATF não é one-size-fits-all. Requer adaptação por stage e resources.

2. Lack of automated tooling

Crítica (DevOps engineer, CSA survey respondent): "ATF é um framework, não um produto. Não existe 'ATF compliance scanner' que eu possa rodar no CI/CD. Cada org tem que build own tooling."

Evidência: ATF fornece specification + implementation guidance, mas não fornece open-source reference implementation ou compliance checker.

Gap: Orgs precisam integrar manualmente ATF checks em pipelines existentes — aumenta time-to-production.

Proposta (community): Create ATF policy-as-code templates (OPA, Sentinel) + GitHub Actions for ATF compliance checks.

Status: Gap reconhecido. MassiveScale.AI (Josh Woodruff's company) oferece commercial tooling, mas sem open-source alternative.

3. Falta de guidance para legacy systems

Crítica (enterprise architect, Financial Services): "ATF assume greenfield deployment. Meu agent precisa integrar com mainframe COBOL system de 1987. Como aplicar ATF Element 4 (Segmentation) quando o legacy system não tem RBAC?"

Evidência: ATF implementation patterns assumem modern cloud infrastructure (JWT, OAuth2, API gateways). Legacy systems requerem workarounds (e.g., proxy layers).

ATF position: "Use API gateway como abstraction layer — legacy system fica behind gateway, agent interage apenas com gateway. Gateway enforces ATF controls."

Avaliação: Solução existe, mas ATF não documenta explicitamente. Gap em guidance para brownfield deployments.

4. Promotion gates podem criar false sense of security

Crítica (AI Safety researcher, anônimo): "Agent que passa por Intern → Junior → Senior → Principal pode AINDA ter failure modes inesperados. 'Zero critical incidents por 8 weeks' não garante ausência de black swan events."

Evidência empírica (não publicada): Agent que operou 6 meses sem incident sofreu catastrophic failure ao encontrar input edge case nunca visto em training/operation.

ATF position: "Promotion gates são necessary but not sufficient. Element 5 (Incident Response) assume que agents IRÃO falhar — circuit breakers e kill switches são failsafes."

Avaliação: Crítica válida. ATF não afirma que Principal agents são 'provably safe' — apenas que são 'sufficiently tested within observed operating conditions'. Black swans são, por definição, undetectable beforehand.

5. Compliance mapping é interpretação, não garantia

Crítica (Legal Counsel, Healthcare): "ATF diz que Element 3 (Data Governance) mapeia para LGPD Art. 46-49 (segurança de dados). Mas meu DPO (Data Protection Officer) discorda. ATF não substitui legal review."

Evidência: ATF compliance mapping table (§4) inclui disclaimer: "Applicability depends on system classification and organizational role". Ou seja, não é one-to-one mapping.

ATF position: "ATF fornece reasonable starting point para compliance — mas orgs DEVEM fazer legal review específico por jurisdiction e use case."

Avaliação: Não é weakness do ATF per se, mas importante lembrar que framework técnico ≠ legal compliance guarantee.

Falhas Empíricas (Onde o ATF Não Se Sustenta)

F1: Behavioral baselines falham em low-volume environments

Context: ATF Element 2 (Behavior) requer "Behavioral Baseline" para anomaly detection.

Falha: Agent que roda 1x por dia durante 2 weeks = 14 data points. Insuficiente para estabelecer baseline estatisticamente significativo.

Evidência: Org que tentou aplicar ATF a batch agent (runs once daily) → anomaly detector flagged 80% das runs como "anomalous" (false positive rate inaceitável).

Workaround: Usar "synthetic baselines" (simular agent em test environment com high volume) ou skip anomaly detection e rely on threshold-based alerts.

F2: Kill switches não funcionam para agents com side effects irreversíveis

Context: ATF Element 5 (Incident Response) requer "Kill Switch — immediate termination <1 second".

Falha: Agent que já enviou email, postou em redes sociais, ou executou transação financeira → kill switch não desfaz ação.

Evidência: Agent de marketing enviou 10,000 emails com conteúdo errado. Kill switch ativado após 2 minutos. Damage already done.

Lesson learned: Element 5 deve incluir "pre-action approval" para irreversible actions (mesmo em Senior/Principal levels) → contradiz goal de autonomia, mas necessário para high-impact actions.

Garantir que o Prontuário, ao entrar em produção, tenha governance baseline para agents que acessarão dados de pacientes. Evitar "shadow agents" e compliance violations desde Day 1.

Escopo (ATF Phase 1 — 2-3 weeks implementation)

ATF Element Implementação Mínima Esforço

2. Behavior Structured logging de todas as queries IMHOTEP faz ao Prontuário → Memory Core 1 day

Total: 10 days (2 weeks) — fit no Sprint 1 do Prontuário (6-8 weeks total).

  • IMHOTEP pode acessar prontuários via API com identity verificável
  • Todas as ações logadas e auditáveis via Memory Core
  • PII não vaza em logs
  • IMHOTEP não pode deletar/modificar prontuários
  • Circuit breaker ativa em caso de anomaly (e.g., flood de requests)

ROI

Custo: 2 weeks dev time (~$3,000 se Daedalus implementar).

Benefício: Evitar 1 compliance violation (LGPD Art. 46 — segurança de dados) = multa potencial de 2% do faturamento anual (R$ 50k-500k dependendo do scale) + dano reputacional. Break-even: se evitar 1 incident em 12 meses, ROI = 16x-166x.

Adaptações Necessárias (ATF → Anima Context)

  1. Maturity timelines: ATF recomenda 2-4-8 weeks. Para Anima (ambiente controlado, single operator Luis), podemos acelerar: 1-2-4 weeks. Compensação: adversarial testing mais rigoroso antes de cada promoção.
  2. Human-in-the-loop: ATF assume multi-stakeholder approval (Security Team, Business Owner, Risk Committee). Anima = Luis é single decision maker. Simplifica governance, mas aumenta risco de single point of failure. Mitigação: Azimute atua como "automated Risk Committee" — flagging decisions que Luis deve revisar.
  3. Agent-to-Agent delegation: ATF não cobre explicitamente (Q1 em §5 Debates). Anima JÁ tem multi-agent (Azimute → Daedalus, Azimute → IMHOTEP). Precisamos de delegation manifest: "Azimute pode delegar implementation para Daedalus SE task ≤ 8h effort AND não envolve dados de pacientes."

[T]
Agentic Trust Framework: Zero Trust for AI Agents (CSA)
https://cloudsecurityalliance.org/blog/2026/02/02/the-agentic-trust-framework-zero-trust-governance-for-ai-agents
Documento-fonte oficial do ATF. Especificação completa dos 5 Core Elements, 4 Maturity Levels, 5 Promotion Gates, e 3 Implementation Phases. Publicado por Josh Woodruff (CSA Research Fellow, Feb 2026). Única fonte primária técnica do framework.
[T]
Agentic AI Security Scoping Matrix (AWS)
https://aws.amazon.com/blogs/security/the-agentic-ai-security-scoping-matrix-a-framework-for-securing-autonomous-ai-systems/
Framework complementar ao ATF. Define 4 scopes de autonomia (No Agency → Prescribed → Supervised → Full Agency) baseados em 2 dimensões: agency (scope de ações permitidas) vs. autonomy (grau de independência). ATF maturity levels mapeiam 1:1 com AWS scopes. Publicado Nov 2025.
[T]
Securing Agentic AI: A Discussion Paper (Singapore CSA + FAR.AI)
https://www.csa.gov.sg/resources/publications/securing-agentic-ai-a-discussion-paper/
Perspectiva governamental sobre agentic AI security. Enfatiza shared responsibility across ecosystem (developers, vendors, enterprises, users, regulators, researchers). Identifica open problems onde investment deve focar. Publicado 2025-2026.
[T]
Governance and Security for AI Agents Across the Organization (Microsoft CAF)
https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ai-agents/governance-security-across-organization
Enterprise governance guidance para Azure AI. Cobre data governance + compliance (GDPR, HIPAA), agent observability (Agent 365, Defender for Cloud), agent security (AI threat protection, red teaming), e agent development (standardize frameworks/protocols). Alinha com ATF principles no contexto Microsoft.
[C]
The AI Agent Identity Crisis: New Research Reveals a Governance Gap (Strata Identity + CSA Survey)
https://www.strata.io/blog/agentic-identity/the-ai-agent-identity-crisis-new-research-reveals-a-governance-gap/
Pesquisa empírica sobre gaps de governança. Survey de 285 IT/security professionals (Sept-Oct 2025). Findings: apenas 18% confiantes que IAM systems atuais podem gerenciar agent identities, 44% usam static API keys, 80% não têm real-time inventory de agents ativos. Evidência de que o problema que ATF resolve é real e widespread.
[A]
NIST AI Risk Management Framework (AI RMF)
https://www.nist.gov/itl/ai-risk-management-framework
Framework governamental de risk management para AI. Define 4 funções: GOVERN, MAP, MEASURE, MANAGE. ATF implementa controles técnicos que cumprem NIST GOVERN 1.1 (agent registration), MAP 1.1 (authentication), MEASURE 2.1 (logging), MANAGE 2.1 (data protection). Referência para compliance mapping.
[T]
MAESTRO Threat Modeling Framework (CSA)
https://cloudsecurityalliance.org/research/working-groups/zero-trust (referência via ATF doc)
Framework de threat modeling complementar ao ATF. MAESTRO identifica riscos em cada camada da arquitetura agentic. ATF fornece os controles de governança que mitigam esses riscos. Relação: MAESTRO (o que pode dar errado?) → ATF (como manter controle?).
[T]
OWASP Top 10 for Agentic Applications (Dec 2025)
https://owasp.org/www-project-top-10-for-large-language-model-applications/ (referência via ATF doc)
Top 10 vulnerabilities em agentic AI. Inclui prompt injection, data poisoning, jailbreak, credential theft. ATF Element 3 (Data Governance) operacionaliza mitigations para essas vulnerabilities. ATF alinha com OWASP guidance como industry standard.
[E]
Agentic AI + Zero Trust: A Guide for Business Leaders (Woodruff, Sept 2025)
Amazon (livro comercial, não open-access)
Deep dive no ATF com foreword de John Kindervag (creator of Zero Trust). Explora implementation patterns detalhados + case studies. Não acessível via web fetch, mas citado no blog post CSA como referência completa.
[T]
NIST 800-207 Zero Trust Architecture (2020)
https://csrc.nist.gov/publications/detail/sp/800-207/final
Codificação oficial dos princípios Zero Trust. "Never trust, always verify" — fundação conceitual que ATF adapta para domínio de agentes autônomos. Referência histórica que contextualiza ATF como application de ZT principles a novo domain.
