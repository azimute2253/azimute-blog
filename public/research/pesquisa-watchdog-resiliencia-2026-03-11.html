🛡️ Watchdog Processes — Resiliência vs Complexidade

Data: 2026-03-11 Origem: Reflection — Night Pipeline Contexto: Blackouts 6h+ no gateway Kerberos

1. Relevância

A pergunta da reflection captura um problema crítico no ecossistema Kerberos: blackouts de 6+ horas no pipeline noturno quando o gateway OpenClaw falha silenciosamente. Atualmente não existe mecanismo automático de recovery — o sistema depende de intervenção manual.

Watchdog timers são um padrão estabelecido em sistemas embarcados e infraestrutura crítica há décadas. A questão central não é se usar watchdog, mas qual configuração mínima elimina blackouts sem introduzir complexidade desnecessária.

Conexão Direta com Kerberos

O gateway atual não possui nem watchdog nem health check robusto. Quando falha (OOM, deadlock, crash silencioso), permanece "up" segundo systemd mas não funcional. Resultado: 6h+ sem heartbeats, compaction, study, backup — pipeline inteiro congelado até intervenção manual.

2. Conceito

Hardware Watchdog

Timer eletrônico independente que monitora o sistema operacional. Se o OS parar de enviar "keepalive" signals dentro do timeout configurado, o hardware força um reset completo. Protege contra kernel panic, systemd failure, e qualquer falha em nível de sistema.

Software Watchdog (systemd)

Protocolo de supervisão onde systemd monitora services e services reportam estado via sd_notify(). Se um service parar de enviar WATCHDOG=1 por mais tempo que WatchdogSec, systemd mata e reinicia o processo.

Process Supervisor ≠ Watchdog

Tools como supervisor, monit, pm2 são process managers, não watchdogs. Reiniciam processos que crasham, mas:

  • Não detectam deadlocks (processo vivo mas travado)
  • Não protegem contra kernel panic ou systemd failure
  • São processos userspace — podem falhar também
⚠️ Anti-pattern: Software-Only Watchdog

"There is no such thing as a software watchdog." Process supervisors são complementos úteis, mas substituir hardware watchdog por supervisor é um erro de design. A cadeia de confiança precisa de um elemento externo que sobreviva a qualquer software failure.

3. Mecanismo

Fluxo Hardware Watchdog

  1. Boot: systemd ativa hardware watchdog (/dev/watchdog) via RuntimeWatchdogSec
  2. Runtime: systemd envia keepalive a cada RuntimeWatchdogSec/2
  3. Failure: Se systemd travar e não enviar keepalive, hardware força reboot após timeout

Fluxo Software Watchdog (systemd → service)

  1. Start: systemd seta env var WATCHDOG_USEC com o timeout em microsegundos
  2. Service lê: WATCHDOG_USEC e calcula intervalo de ping (geralmente timeout/2)
  3. Loop: Service envia sd_notify("WATCHDOG=1") via Unix socket ($NOTIFY_SOCKET)
  4. Failure: Se systemd não receber ping por WatchdogSec, mata o processo (SIGABRT) e reinicia conforme Restart=

Configuração Mínima (systemd)

# /etc/systemd/system.conf
[Manager]
RuntimeWatchdogSec=30s
ShutdownWatchdogSec=10min

# /etc/systemd/system/openclaw-gateway.service
[Unit]
Description=OpenClaw Gateway
After=network.target

[Service]
Type=notify
ExecStart=/usr/local/bin/openclaw gateway start
WatchdogSec=60s
Restart=on-failure
RestartSec=5s
StartLimitBurst=3
StartLimitIntervalSec=300

[Install]
WantedBy=multi-user.target

Implementação (Node.js/Bash)

Node.js não tem binding nativo pra sd_notify. Opções:

  1. NPM package sd-notify: Wrapper leve pra protocol
  2. Bash wrapper: Script que monitora processo Node + envia notify
  3. systemd-notify CLI: Chamar systemd-notify WATCHDOG=1 via child_process
// Opção 1: sd-notify package
const notify = require('sd-notify');
const WATCHDOG_USEC = parseInt(process.env.WATCHDOG_USEC || 0);

if (WATCHDOG_USEC > 0) {
  const interval = WATCHDOG_USEC / 2000; // metade do timeout, em ms
  setInterval(() => {
    notify.watchdog();
  }, interval);
}

notify.ready(); // Informa systemd que service tá pronto

4. Estado da Arte

Multi-Stage Watchdogs

Sistemas críticos (aerospace, medical devices) usam watchdogs em cascata:

  1. L1 — Application: Thread interna monitora event loop
  2. L2 — Process supervisor: Reinicia processo se crashar
  3. L3 — systemd watchdog: Reinicia service se deadlock
  4. L4 — Hardware watchdog: Reboot se systemd falhar

Cloud-Native: Kubernetes Liveness Probes

Kubernetes implementa conceito similar com livenessProbe (HTTP/TCP/exec check). Se falhar N vezes consecutivas, kubelet mata o container. Diferença crucial: probe é pull (k8s pergunta), watchdog é push (service informa).

Systemd Watchdog (2026)

Artigo da OneUpTime (março 2026) confirma que padrão atual é:

  • Type=notify obrigatório
  • WatchdogSec entre 10s-120s dependendo da criticidade
  • Restart=on-failure (não on-watchdog, que é subset)
  • StartLimitBurst=3-5 pra evitar restart loops

5. Debates & Controvérsias

systemd Watchdog vs External Monitoring

Argumento pró-external: Tools como Monit, Nagios, Prometheus+Alertmanager são mais flexíveis — monitoram múltiplos hosts, integram com dashboards, alertas customizados.

Argumento pró-systemd: External monitoring tem latência inerente (polling interval + network delay). systemd watchdog é local, determinístico, e não depende de rede. Complementares, não excludentes.

Timeout Auctions

Artigo de Alexandru Lazar (embeddedrelated.com) identifica "timeout auctions" como anti-pattern: escolher timeout arbitrariamente ("30s parece bom") sem análise técnica. Resultado: ou falsos positivos (timeout muito curto) ou detecção tardia (timeout longo demais).

Recomendação: Basear timeout em dados reais — tempo máximo de operação legítima + margem de segurança. Exemplo: se event loop do gateway processa batch em max 5s, WatchdogSec=15s é razoável.

Fire-and-Forget Anti-pattern

Comum em código didático mal escrito: feed watchdog incondicionalmente via timer interrupt. Resultado: watchdog sempre feliz mesmo se application loop travou. Solução: vincular watchdog reset ao progresso real da aplicação (ex: só fazer ping após processar fila de mensagens).

6. Análise Crítica

Limitações do Watchdog

  • Não previne bugs: Watchdog detecta e recupera de falhas, mas não elimina a causa raiz. Se service crashar por bug, watchdog vai reiniciar indefinidamente até StartLimitBurst.
  • Janela de downtime: Mesmo com timeout agressivo (10s), há 10s de indisponibilidade. Para sistemas 99.999% uptime, isso pode ser inaceitável.
  • State loss: Restart implica perda de estado in-memory (a menos que exista checkpoint/recovery mechanism).

Hardware Watchdog: Nem Sempre Disponível

Alguns sistemas (VMs, containers, certos laptops) não expõem /dev/watchdog. Verificar com: 

wdctl /dev/watchdog0
# ou
dmesg | grep -i watchdog

Se hardware watchdog indisponível: software watchdog ainda vale a pena. Protege contra service-level failures (deadlocks, memory leaks) mesmo sem proteção contra kernel panic.

Complexidade vs Resiliência

Configuração mínima (systemd unit + sd_notify) adiciona ~20 linhas de código e ~10 linhas de config. ROI é alto pra sistemas críticos. Mas: se service é não-crítico e downtime aceitável, watchdog pode ser overkill. Exemplo: ferramenta CLI de uso único não precisa de watchdog.

7. Intersecção com Kerberos

Diagnóstico Atual

Componente Status Atual Failure Mode Recovery
OpenClaw Gateway Sem watchdog Crash silencioso, deadlock, OOM Manual (systemctl restart)
Night Pipeline Depende do gateway Blackout 6h+ se gateway falha Manual (restart gateway + re-trigger cron)
Heartbeat System Health check básico Detecta falha mas não recupera Alerta (Telegram) → intervenção manual

Solução Proposta

✅ Configuração Mínima (Fase 1)
  1. Ativar hardware watchdog: RuntimeWatchdogSec=30s em /etc/systemd/system.conf
  2. Converter gateway pra systemd unit: Type=notify, WatchdogSec=60s
  3. Implementar sd_notify no gateway: Usar package sd-notify (NPM) ou wrapper bash
  4. Testar failure modes: Simular deadlock (infinite loop), OOM, crash → confirmar restart automático

Benefícios Imediatos

  • Zero blackouts >60s: Se gateway travar, systemd reinicia em <60s
  • Proteção kernel-level: Se systemd travar, hardware reboot em <30s
  • Baixa complexidade: ~30 linhas de código + config, zero deps externas pesadas
  • Padrão replicável: Template pra serviços da agência (clientes com infraestrutura crítica)

Fase 2 (Opcional): Multi-Stage

  • L1: Internal health check (gateway monitora próprio event loop)
  • L2: systemd watchdog (atual)
  • L3: External monitoring (Prometheus + Alertmanager) pra métricas + alertas
