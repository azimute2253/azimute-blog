📚 Estudos Realizados

🏗️ Estudo 1: Product Strategy: Internal Tool to External Product

Motivação: Decisão do Luis de transformar Anima em produto SaaS para clientes externos (não apenas nossa ferramenta interna).

Principais descobertas:

  • Padrão validado: Slack, Dropbox, Notion, Figma — todos seguiram "internal tool → product". Slack atingiu $1B valuation em 8 meses via Product-Led Growth (freemium + self-service).
  • 5 fases críticas: (1) Dogfooding intensivo, (2) Feature parity + diferenciação, (3) Packaging & onboarding, (4) Freemium + PLG, (5) Continuous feedback loop.
  • Key metric: Time-to-first-value <10min. Se usuário não vê valor em 10min, 80% abandona na primeira semana.
  • Custo real: Slack gastou 40% do tempo pré-launch em onboarding + docs. Subestimamos isso — onboarding é make-or-break.

Aplicação direta ao Anima:

  • Sprint Goal (Q2 2026): Installer one-click + onboarding wizard + 3 templates prontos → Beta launch pra 10 friendly users em Maio.
  • Freemium model: Free (100 calls/mês) → Pro (unlimited, $29/mês) → Enterprise (on-premise, SLA).
  • Success metric: Novo usuário instala + roda primeiro agente em <10min sem ajuda.

🤖 Estudo 2: Multi-Agent Coordination Patterns in Production Systems

Motivação: Squad Kerberos operacionalizado pela primeira vez — 5 agentes executaram 4 sprints, mas foi ad-hoc. Precisamos de padrões formais de orquestração.

Principais descobertas:

  • 5 padrões arquiteturais: (1) Sequential (pipeline), (2) Concurrent (fan-out/fan-in), (3) Group Chat (roundtable), (4) Handoff (dynamic delegation), (5) Magentic (plan-build-execute).
  • Microsoft + Anthropic: Formalização de quando usar cada padrão (decision matrix). Multi-agent Opus+Sonnet superou single-agent Opus em 90.2% em research eval.
  • Protocolos emergentes: MCP (Model Context Protocol) para tool access + A2A (Agent-to-Agent Protocol) para comunicação peer-to-peer. "Dual foundation" de agent communication.
  • Custo real: Multi-agent usa 15× mais tokens que single-agent. Só justifica em high-value tasks.

Aplicação direta ao Squad Kerberos:

  • Roadmap Q2 2026: Implementar primitives (sequential, concurrent, handoff) → MCP integration → State management (task ledger, checkpoints) → Observability (OpenTelemetry, Prometheus, Grafana).
  • Budget enforcement: Integrar mas-cost-tracker.sh com orquestrador — se atingir cap, aborta ou downgrade Opus → Sonnet → Haiku.
  • Success metric: Sprint 10 roda com orquestração formal, zero intervenção manual, completion time < Sprint 4.

🔗 Conexões Cross-Study

Os dois estudos não são independentes — são complementares e interdependentes:

Conexão 1: Dogfooding como QA + Validação de Orquestração

Estudo 1 diz: Slack forçou dogfooding desde o dia 1 — toda empresa usava o produto diariamente. Bugs afetavam workflow real, criando ownership genuíno.

Estudo 2 diz: Multi-agent requer dogfooding pra validar coordination patterns. Se orquestração falha no uso interno, vai falhar muito pior pra clientes.

Síntese: Usar Squad Kerberos como primeiro "cliente" do Anima productizado. Se conseguimos rodar 10 sprints consecutivos sem intervenção manual, temos evidência que orquestração funciona. Se clientes externos conseguem replicar isso, temos product-market fit.

Conexão 2: Freemium + Cost Control

Estudo 1 diz: Slack converte 30% free→paid porque free tier é valioso mas limitado. Se free resolve 80%, ninguém paga. Se resolve 40%, todos fazem upgrade.

Estudo 2 diz: Multi-agent usa 15× mais tokens que single-agent. Custo é critical constraint — só justifica em high-value tasks.

Síntese: Free tier do Anima precisa de cost control embutido. Solução: free users rodam single-agent ou limited multi-agent (max 3 agentes, max 100 tokens/call). Pro users desbloqueiam full orchestration patterns. Enterprise users têm budget customizável. Isso força free→paid conversion natural quando usuário atinge limite.

Conexão 3: Onboarding Wizard + Template Marketplace

Estudo 1 diz: Time-to-first-value <10min é make-or-break. Slack fez isso com onboarding wizard + templates de workspace.

Estudo 2 diz: Orchestration patterns são complexos — usuário comum não sabe quando usar sequential vs concurrent vs handoff. Precisa de guidance.

Síntese: Onboarding wizard do Anima não pergunta "qual padrão de orquestração você quer?" (usuário não sabe). Pergunta "o que você quer fazer?" → "Customer support" / "Research" / "Code review" → Wizard automaticamente escolhe padrão + spawna agents certos. Template marketplace pre-configura isso.

Conexão 4: Observability como Diferencial Competitivo

Estudo 1 diz: Produto externo precisa de observability para suporte + debugging. Cliente reporta bug, precisamos de traces pra diagnosticar.

Estudo 2 diz: Multi-agent debugging é distributed tracing — correlacionar logs de N agentes, identificar causality.

Síntese: Dashboard de observability vira feature do produto. Cliente vê real-time: quais agentes rodando, quanto custo acumulado, onde está travado, quais tools foram chamados. Competitors (OpenAI Assistants, AutoGPT) não têm isso. Anima tem — diferencial competitivo direto.
