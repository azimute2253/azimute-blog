LLM Routing: Automatic Model Selection Based on Task Complexity

1. Relevância

O problema de routing automático entre modelos é crítico para sistemas multi-agente como o Kerberos. A reflection identificou que "os agentes estão todos usando Opus" — um padrão de desperdício onde queries simples consomem o modelo mais caro/capaz desnecessariamente.

Conexão direta: O Kerberos usa PE-Cascade (L1→L9) para classificar complexidade de tarefas, mas não implementa routing automático de modelos. A pesquisa revela que sistemas em produção economizam 30-84% de custo sem perder qualidade quando roteiam dinamicamente queries para modelos apropriados.

Segundo o survey arXiv de 2026 sobre Dynamic Model Routing, o problema fundamental é:

"Production deployment of large language models faces a fundamental cost-performance dilemma. Queries vary substantially in complexity, from simple factual questions to complex multi-step reasoning problems. When a single model handles all requests, simple queries consume unnecessary resources when routed to powerful models, while complex queries may exceed the capabilities of smaller models."

2. Conceito

LLM routing é a decisão lógica que direciona requests para modelos apropriados baseado nas características do request. Não é sobre o mecanismo de routing em si, mas sobre entender o trabalho bem o suficiente para categorizá-lo significativamente.

Tipos de Routing

  • Static Routing: UI components distintos para tarefas específicas (ex: interface de marketing vs análise de negócios). Simples, mas inflexível.
  • Dynamic Routing: Layer de decisão intercepta requests e roteia para o modelo downstream mais adequado. Essencial para chatbots, assistentes virtuais, e sistemas multi-propósito.

Quando Routing é Necessário

Você precisa de routing quando tem um ou mais destes problemas:

  1. Custos escalando mais rápido que receita (margens encolhendo)
  2. Latência imprevisível afetando conversão
  3. Casos de uso obviamente diferentes usando o mesmo modelo
  4. Outage de provider derrubando o produto inteiro
  5. Evitando features porque custo de LLM seria proibitivo
Regra de ouro: Se você processa menos de 1000 requests/dia com custo < $300/mês, routing é desperdício de tempo. Seu tempo vale mais que a economia. Use um bom modelo e siga em frente.

3. Mecanismo

Paradigmas de Routing (Survey arXiv 2026)

Existem 6 paradigmas principais de routing, não mutuamente exclusivos:

Paradigma Descrição Exemplos
Difficulty-Aware Roteia baseado em complexidade estimada da query BEST-Route, vLLM Semantic Router, GraphRouter
Preference-Aligned Usa dados de preferência humana para treinar routers RouteLLM, Arch-Router, P2L, Eagle
Clustering-Based Agrupa queries similares via aprendizado não-supervisionado UniRoute, Avengers-Pro
Reinforcement Learning Aprende políticas de routing via feedback online Router-R1, R2-Reasoner, MetaLLM, MixLLM, PILOT
Uncertainty-Based Roteia baseado em confiança do modelo CP-Router, probe-based methods, LLM-as-judge
Cascading Sequencialmente tenta modelo pequeno → escala se necessário AutoMix, FrugalGPT, Self-REF, LM-Blender

Fatores-Chave para Decisões de Routing

1. Custo e Volume

Queries variam em valor de negócio. Um chatbot de suporte respondendo "Onde está meu pedido?" tem baixo valor por query (economia de $5 em ticket humano). Já um assistente de escrita criativa gerando copy para cliente enterprise ($500/mês) justifica modelos caros mesmo que custem 100× mais.

Lógica de routing: alto-volume + baixo-valor → modelo barato | baixo-volume + alto-valor → modelo caro

2. Latência e UX

  • Requests interativos (autocomplete, chat, inline code): precisam latência sub-segundo (< 300ms). Otimize para velocidade mesmo que custe mais ou tenha qualidade ligeiramente menor.
  • Requests background (processamento de documentos, resumos noturnos): podem ser lentos. Otimize para custo e qualidade.

3. Complexidade da Tarefa

Proxies comuns para complexidade:

  • Comprimento do input: inputs longos geralmente requerem modelos mais capazes
  • Keywords: "analyze", "compare", "evaluate" → alta complexidade | "extract", "classify", "detect" → baixa complexidade
  • Classifiers treinados: rotular historical requests como "simple" vs "complex" e treinar modelo leve para prever (80% de acurácia já economiza muito)

4. Privacy e Compliance

Dados sensíveis criam hard constraints que sobrescrevem custo/performance:

  • HIPAA → providers com Business Associate Agreements
  • Dados financeiros → providers que não treinam em inputs
  • Dados EU → data centers europeus

Padrões que Funcionam em Produção

Rule-Based Routing (80% dos casos)

if input_length < 100:
    route_to("gpt-3.5-turbo")
elif user_tier == "free":
    route_to("claude-haiku")
elif region == "EU":
    route_to("eu-hosted-model")
else:
    route_to("claude-opus")

Vantagens: clareza (trace exato de decisões), manutenibilidade (centralizar regras em config), 80% das necessidades cobertas com 5-10 regras simples.

Confidence-Based Routing

Tenta modelo rápido primeiro. Se confiança baixa (< 95%), escala para modelo mais capaz.

Economia: Se 70% são resolvidos com confiança pelo modelo barato, reduz uso do modelo caro em 70% mantendo qualidade.

Fallback Chains

primary_model → backup_model → tertiary_model → fail

Circuit breakers: se modelo está timing out na maioria dos requests, para de enviar traffic temporariamente.

5. Debates e Controvérsias

User Choice vs Automatic Routing

CodeRabbit argumenta que "choice is not always good" — usuários escolhendo modelos manualmente leva a:

  • Inconsistência (cada user com preferências diferentes)
  • Desperdício (users escolhendo modelos caros por padrão sem necessidade)
  • Complexidade cognitiva (usuários não são experts em trade-offs de modelo)

Posição majoritária: routing automático com override manual quando necessário (power users, debugging).

Pre-Generation vs Post-Generation Routing

  • Pre-generation: decide antes de gerar output (baseado em propriedades da query). Mais rápido, mas menos informado.
  • Post-generation: gera resposta inicial, avalia qualidade/confiança, decide se re-roteia. Mais preciso, mas adiciona latência.

AutoMix demonstra que self-verification (post-gen) via few-shot é confiável o suficiente para cascading, mas estudos anteriores (Huang 2024, Mahaut 2024) mostraram que self-verification falha para reparar outputs. Consenso: self-verification funciona para routing, não para correction.

Classifier LLM vs Embeddings

LLM-assisted routing: usa LLM classificador (ex: Titan Text) para categorizar query → adiciona custo + latência, mas oferece fine-grained classification.

Semantic routing: usa embeddings (ex: Titan Embeddings V2) + similarity search em FAISS → mais rápido (100ms vs 500ms), mas coarse-grained.

AWS blog mostra análise de custo real:

Método Custo/mês (100k queries) Latência Média
LLM-assisted $188.90 ~600ms (classification 500ms + answer 100-2700ms)
Semantic $107.90 ~300ms (embedding 100ms + answer 200-2700ms)

Economia potencial: usar Claude Haiku ($618/mês) vs Claude Opus ($7425/mês) para 50% das queries economiza ~$3400/mês — custo de routing é desprezível comparado.

6. Crítica e Limitações

1. Over-Engineering Prematuro

LogRocket adverte: "The first time your LLM bill crosses $10,000 in a month, you start paying attention." Routing em $500/mês é waste. Maioria das startups não precisa até atingir volume significativo.

2. Calibration de Thresholds

Confidence-based routing requer calibrar thresholds de confiança:

  • Threshold alto demais → over-routing para modelos caros
  • Threshold baixo demais → serve respostas ruins

Requer experimentação iterativa + monitoring contínuo. Não há fórmula mágica.

3. Generalização para Modelos Novos

Métodos como Zooter (reward-guided) e EmbedLLM (matrix factorization) treinam em LLMs fixos → não generalizam para novos modelos sem retraining.

Soluções:

  • UniRoute: usa clustering + validation set, pode adicionar novos LLMs sem retrain (apenas evaluar em clusters existentes)
  • GraphRouter: GNN aprende patterns de node characteristics, não modelos específicos → inductivo para novos LLMs

4. Custo de Observability

Routing introduz novos modos de falha. Sem logs detalhados (routing decision, matched rules, latency breakdown), debugging é guesswork. Distributed tracing (OpenTelemetry) + cost tracking per route são essenciais mas adicionam overhead.

5. Cold Start em Semantic Routing

Reference prompt set precisa cobrir todos os casos de uso. Se não tem exemplos suficientes de determinado tipo de query, similarity search falha. Requer coleta contínua de prompts reais + iteração.

7. Intersecção com Kerberos e 9 Neurônios

PE-Cascade como Routing Foundation

O Kerberos já implementa classificação de complexidade via PE-Cascade (L1→L9), mas não faz routing automático de modelos. Atualmente:

  • L1-L3 (simple input) → resolvido por sonnet mas poderia ser haiku
  • L4-L6 (genuine agency) → roteado para sonnet mas alguns casos requerem opus
  • L7-L9 (self-awareness) → sempre opus, mas nem todos L7+ precisam do modelo mais caro
Oportunidade: Usar delegation tier do PE-Cascade como input para routing automático:
  • L1-L3 + curto (< 500 tokens) → haiku
  • L4-L6 + médio (500-2000 tokens) → sonnet
  • L7-L9 ou longo (> 2000 tokens) → opus

Squad Routing (Azimute → Squad Members)

Atualmente o orchestrator recebe tudo em opus. Poderia implementar rule-based routing:

if task_type == "PRD review":
    route_to(azimute, model="sonnet")  # análise de docs
elif task_type == "implement backend":
    route_to(delta, model="opus")       # code generation complexo
elif task_type == "QA validation":
    route_to(sigma, model="haiku")      # checklist execution
elif task_type == "ADR creation":
    route_to(azimute, model="opus")     # architectural decisions

Multimodal Routing para 9NN

A teoria dos 9 Neurônios postula camadas sensório-motoras (L1-L3) e cognitivas superiores (L4-L9). LLMRouter demonstra routing multimodal (Video/Image + Text) em Geometry3K, MathVista, Charades-Ego.

Aplicação futura: quando o modelo próprio 500B+ tiver capacidades multimodais, routing poderia distinguir:

  • Input visual simples (object detection) → módulo L1-L2 especializado
  • Input visual + reasoning (scene understanding) → L4-L6 integrado
  • Input multimodal + meta-cognição (avaliar própria percepção) → L7-L9 full

Autonomous Work Queue Integration

Work queue do Kerberos poderia incluir metadata de complexidade estimada. Quando agente faz work_queue(action="pick"), sistema poderia:

  1. Estimar complexidade da task via embedding similarity com historical tasks
  2. Rotear automaticamente para modelo apropriado
  3. Checkpoints salvam tentativas com modelo mais fraco que falharam → escalação automática
