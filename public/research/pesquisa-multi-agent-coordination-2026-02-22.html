Custo de NÃO saber isso:

§2. Conceito Fundamental

Definição

Um Multi-Agent System (MAS) é uma arquitetura onde múltiplas instâncias de LLMs operam com contextos de conversação separados, coordenados por código. Diferente de um single-agent com múltiplos módulos, cada agente em um MAS é uma entidade autônoma com:

  • Percepção própria: Sensors sobre o ambiente
  • Ação própria: Effectors para modificar o ambiente
  • Objetivos: Goals que podem ou não alinhar com outros agentes
  • Autonomia: Capacidade de decidir sem controle centralizado absoluto

Coordination patterns definem como esses agentes compartilham informação, delegam tarefas e sincronizam execução para atingir objetivos coletivos.

Contexto Histórico

A pesquisa em MAS tem raízes em três tradições que convergiram nos anos 1980-1990:

  • Distributed AI (1980s): Lesser & Corkill — como decompor problemas complexos em redes cooperativas
  • Agent-based modeling: Schelling (1971), Epstein & Axtell (1996) — fenômenos macro emergem de interações micro
  • Blackboard architecture (1985): Hayes-Roth — especialistas humanos colaborando em torno de um quadro compartilhado

Arquiteturas clássicas:

  • BDI (Belief-Desire-Intention): Agentes baseados em crenças, desejos e intenções
  • Contract Net Protocol: Negociação por leilão para alocação de tarefas
  • Blackboard Systems: Espaço compartilhado onde knowledge sources contribuem até convergir em solução
  • FIPA-ACL/KQML: Agent Communication Languages para interação padronizada

Evolução Moderna (2023-2026)

A explosão de LLMs trouxe três mudanças fundamentais:

  1. Natural Language Coordination: Agentes podem se comunicar em linguagem natural, não apenas protocolos formais
  2. Flexible Role Assignment: Papéis definidos por prompts, não hard-coded
  3. Emergent Collaboration: Padrões de coordenação que surgem da interação, não apenas regras pré-definidas
Virada Conceitual (2025):
Google DeepMind prova que MAS tem scaling laws quantificáveis — não é heurística, é ciência. Anthropic publica "when NOT to use multi-agent", quebrando o mito "more agents = better".

§3. Mecanismo — Como Funcionam os Padrões de Coordenação

Taxonomia de Topologias

Existem 5 topologias principais, cada uma com trade-offs diferentes:

1. Single-Agent System (SAS)

Um agente único com múltiplos módulos. Mínima coordenação, mas limitado em paralelização. 

SAS: [Planning → Acting → Reflection] → Output
Overhead: Baixo | Parallelism: Zero | Error amplification: 1x

2. Independent MAS

Múltiplos agentes trabalham em paralelo, sem comunicação. Outputs são sintetizados no final. 

Agent1 → Result1 ↘
Agent2 → Result2 → Synthesizer → Final Answer
Agent3 → Result3 ↗
Overhead: Baixo | Parallelism: Alto | Error amplification: 17.2x (!)
⚠️ The 17x Error Trap:
Sem mecanismo de verificação mútua, erros se propagam sem correção. Um agente alucina, outros não detectam, o synthesizer consolida o erro. Google mediu: 17.2x error amplification em topologias independentes.

3. Centralized MAS (Hub-and-Spoke)

Um Orchestrator coordena workers especializados. Circuit breaker natural. 

         ┌─> Worker1 (Expert A)
         │
Orchestrator ─┼─> Worker2 (Expert B) ─> Orchestrator → Final
         │
         └─> Worker3 (Expert C)
         
Error amplification: 4.4x (contido pelo orchestrator)

Vantagens: Decisões globais otimizadas, circuit breaker, accountability clara
Desvantagens: Single point of failure, bottleneck em high-scale, autonomia reduzida

4. Decentralized MAS (Peer-to-Peer Debate)

Agentes debatem em múltiplos rounds, decidem por maioria. 

Round 1: A → B → C → A (debate)
Round 2: A → B → C → A (refine)
Round 3: Majority Vote → Decision

Communication: n² | Token cost: Alto | Robustness: Média

Quando funciona: Tarefas onde diversidade de perspectiva importa (brainstorming, crítica)
Quando falha: Tarefas sequenciais, high tool-count (coordination tax explode)

5. Hybrid MAS

Orchestrator central + peer-to-peer seletivo. Mais flexível, mais complexo. 

Orchestrator
    ↓
Worker1 ↔ Worker2 (peer messaging permitido)
    ↓
Orchestrator (synthesis)
    
Complexity: Alta | Flexibility: Alta | Maintenance: Alta

Blackboard Architecture (Especial)

Todos os agentes leem/escrevem em um espaço compartilhado. Control Unit seleciona quem age. 

┌─────────────────────────────┐
│      BLACKBOARD             │
│  ┌──────┬──────┬──────┐    │
│  │ Msg1 │ Msg2 │ Msg3 │    │
│  └──────┴──────┴──────┘    │
└─────────────────────────────┘
       ↑         ↓
   Control Unit (selects next agent)
       ↓
   Agent Pool: [Planner, Executor, Critic, Cleaner, Decider]

Inovação (Han et al., 2025): Blackboard em LLM MAS reduz token cost porque blackboard substitui memory modules individuais. Todos os agentes compartilham o mesmo contexto histórico via blackboard, em vez de cada um manter memória separada.

Key Insight — Topology as Safety Feature:
Centralização NÃO é apenas sobre eficiência — é sobre conter erro. Orchestrator atua como "validation bottleneck", pegando erros antes de propagarem. Independent MAS é rápido mas perigoso. Centralized é devagar mas robusto.

§4. Estado da Arte (2024-2026)

Avanços Recentes

1. Model Context Protocol (MCP) — Anthropic, 2024

Protocolo padrão para conectar LLMs a fontes de dados e ferramentas. Resolve o "disconnected models problem" ao fornecer interfaces unificadas para:

  • Prompts: Templates compartilháveis
  • Resources: Documentos/dados estruturados
  • Tools: Funções executáveis
  • Sampling: Servidores podem pedir ao modelo para gerar completions

Impacto em MAS: Permite context sharing padronizado entre agentes, reduzindo overhead de coordenação. Arquitetura cliente-servidor facilita implementação de topologias híbridas.

2. Scaling Laws of Agency — Google DeepMind, Dez 2025

Estudo com 180 configurações de agentes across 4 benchmarks (Finance-Agent, BrowseComp-Plus, PlanCraft, Workbench). Primeira quantificação rigorosa de quando MAS funciona.

Principais descobertas:

a) The Alignment Principle

Tarefas paralelizáveis (análise financeira com múltiplos aspectos independentes) beneficiam massivamente de Centralized MAS: +80.9% over SAS.

b) The Sequential Penalty

Tarefas sequenciais (planejamento com dependências de estado) degradam com qualquer MAS: -39% a -70%. Overhead de comunicação fragmenta a cadeia lógica.

c) The Tool-Coordination Tradeoff

Conforme o número de tools cresce (16+), o "tax" de coordenar múltiplos agentes cresce desproporcionalmente. Single-agent com Tool Search Tool pode ser melhor.

d) The 45% Saturation Point

Multi-agent coordination tem maior ROI quando single-agent baseline está < 45% accuracy. Se SAS já atinge 80%, adicionar agentes introduz mais ruído que valor.

e) Predictive Model (R² = 0.513)

Google treinou um modelo que, dado task properties (sequential dependencies, tool density, decomposability), prediz a topologia ótima com 87% de acerto em configs não vistas.

3. When NOT to Use Multi-Agent — Anthropic, 2025

Anthropic publica guidelines explícitas baseadas em produção:

  • Context pollution degrades performance: Quando subtasks adicionam contexto irrelevante ao main task
  • Tasks run in parallel: Quando search space é grande e pode ser explorado independentemente
  • Specialization improves tool selection: Quando 20+ tools causam confusion em single-agent

FORA dessas situações, single-agent é melhor. Token cost de MAS: 3-10x maior que SAS para tarefa equivalente.

4. Blackboard-based LLM MAS (LbMAS) — Han et al., 2025

Primeira implementação de blackboard architecture em LLM MAS. Innovations:

  • Dynamic agent selection: Control Unit escolhe agentes com base no conteúdo atual do blackboard
  • Shared memory via blackboard: Elimina memory modules individuais, reduz token cost
  • Public + Private spaces: Permite debates privados sem poluir contexto global

Performance: Competitivo com SOTA (GPTSwarm, AFlow, MaAS) usando metade dos tokens (4.7M vs 13M no MATH benchmark). Autonomous sem training prévio.

Tendência 2026:
Shift de "how to build MAS" para "when NOT to build MAS". Modelos cada vez mais capazes (Opus 4.5, GPT-5.2) resolvem sozinhos o que antes precisava de squad. MAS virará solução de nicho para problemas genuinamente complexos e paralelizáveis.

§5. Debates Existentes

Controvérsia 1: "More Agents Is All You Need"

Posição A (2024): Papers como "More Agents Is All You Need" (Wang et al., 2024) e "Collaborative Scaling" (Chen et al., 2024) argumentam que performance escala com agent count.

Contra-argumento (Google, 2025): Estudos controlados provam que isso é verdade apenas em condições específicas:

  • Tarefa altamente paralelizável (Finance-Agent: +80%)
  • Single-agent baseline fraco (<45%)
  • Topologia correta (Centralized > Independent)

Em tarefas sequenciais (PlanCraft), every MAS degraded -39% to -70%. O debate foi resolvido empiricamente: "it depends on task properties."

Controvérsia 2: Centralization vs Decentralization

Descentralização (advocacy):

  • Maior robustez (no single point of failure)
  • Melhor escalabilidade
  • Preserva autonomia dos agentes

Centralização (advocacy):

  • Contenção de erro (4.4x vs 17.2x amplification)
  • Otimização global possível
  • Circuit breaker natural

Estado do Debate: Não há vencedor universal. Depende de error tolerance do domínio:

  • Medical/Financial systems: Centralização é mandatória (erro = morte/perda $)
  • Creative/Research systems: Decentralização pode valer (diversidade > precisão)

Controvérsia 3: Blackboard vs Message-Passing

Message-Passing (MetaGPT, AutoGen): Agentes se comunicam diretamente via mensagens. Explícito, rastreável.

Blackboard (LbMAS): Agentes leem/escrevem em espaço compartilhado. Comunicação implícita via estado compartilhado.

Trade-off:

  • Message-passing: Mais controle, mais overhead (cada mensagem = contexto duplicado)
  • Blackboard: Menos overhead (contexto único), mas coordenação mais complexa (precisa de Control Unit sofisticado)

Evidência empírica: LbMAS usou 64% menos tokens que MaAS (message-passing) no MATH benchmark. Blackboard pode ser superior em token efficiency.

Controvérsia 4: Heterogeneous vs Homogeneous Teams

Heterogeneous (modelos diferentes por papel):

  • Anthropic: Weak orchestrator + strong workers beats all-strong (+31% em BrowseComp-Plus)
  • OpenAI/Gemini: Degrada performance (orchestrator bottleneck)

Homogeneous (mesmo modelo):

  • Mais previsível, menos overhead de configuração
  • Pode ser sub-ótimo se modelo não é bom em todos os papéis

Consensus emergente: "Spend on workers, not manager" — sub-agent capability matters more than orchestrator capability. Mas validar por model family.

Questão NÃO Resolvida:
Ninguém ainda quantificou quando usar Hybrid vs Centralized. Google testou ambos, mas não forneceu decision tree claro. Cursor (2026) relata que Hybrid funciona para coding agents, mas não generaliza para outros domínios.

§6. Pontos de Crítica — Limitações com Citação

1. Token Explosion (3-10x overhead)

Fonte: Anthropic, "Building Multi-Agent Systems" (2025)
Evidência: Multi-agent systems typically use 3-10x more tokens than single-agent for equivalent tasks due to:

  • Duplicating context across agents
  • Coordination messages between agents
  • Summarizing results for handoffs

Implicação: Custo operacional pode tornar MAS inviável para aplicações de alto volume.

2. Latency Increase (não é mais rápido, é mais lento)

Fonte: Anthropic, "Building Multi-Agent Systems" (2025)
Contra-intuição: "While parallelism helps reduce total execution time compared to running all that work sequentially, multi-agent systems often take longer overall than single-agent systems because of the sheer increase in total computation."

Razão: Paralelização reduz wall-clock time PER subtask, mas total computation (soma de todos os tokens) aumenta. Se API throttling existe, latência pode até piorar.

3. The Early Victory Problem (verificação superficial)

Fonte: Anthropic, "Verification Subagent Pattern" (2025)
Falha comum: Verification agents rodam 1-2 testes, veem passar, declaram sucesso. Não fazem comprehensive validation.

Mitigação: Instruções explícitas tipo "You MUST run the complete test suite before marking as passed" são essenciais.

4. Context-Centric vs Problem-Centric Decomposition

Fonte: Anthropic, "Context-Centric Decomposition" (2025)
Erro comum: Dividir por tipo de trabalho (Planner, Executor, Reviewer) em vez de por fronteiras de contexto.

Resultado: "Telephone game" — cada handoff perde fidelidade. Agents gastam mais tokens coordenando que executando.

Regra correta: Dividir apenas quando contexto pode ser verdadeiramente isolado (ex: frontend vs backend com API bem definida).

5. Capability Saturation (modelos fortes não precisam de MAS)

Fonte: Google DeepMind, "Towards a Science of Scaling Agent Systems" (2025)
Evidência: "The 45% rule" — multi-agent coordination yields highest returns when single-agent baseline is below ~45%. When base model already hits 80%, adding agents introduces more noise than value.

Implicação: Conforme modelos evoluem (Opus 4.5, GPT-5.2), menos problemas justificam MAS. MAS pode ser solução temporária para limitações de modelos atuais.

6. Lack of Theoretical Framework (ainda é heurística)

Fonte: Han et al., "Exploring Advanced LLM MAS Based on Blackboard Architecture" (2025)
Limitação reconhecida: "This is the first attempt to combine blackboard architecture with LLM multi-agent systems, and there are limitations in the implementation."

Mesmo com o modelo preditivo do Google (R²=0.513), ainda há ~50% de variância não explicada. Decisões de arquitetura ainda dependem muito de trial-and-error.

⚠️ Viés Metodológico Crítico:
Todos os estudos (Google, Anthropic, Han) testaram em benchmarks acadêmicos (MATH, GPQA, BrowseComp). Nenhum testou em production systems de larga escala com real users. Generalização para mundo real não é garantida.

Proposta 1: Adaptive Topology Selection

Implementar scripts/topology-router.sh que, dado um task, prediz topologia ótima: 

#!/bin/bash
# topology-router.sh — Adaptive MAS Topology Selection

TASK_DESC="$1"

# Classify task properties
PARALLELIZABLE=$(detect_parallel_subtasks "$TASK_DESC")
SEQUENTIAL=$(detect_dependencies "$TASK_DESC")
TOOL_COUNT=$(count_required_tools "$TASK_DESC")

if [ "$SEQUENTIAL" == "high" ]; then
    echo "TOPOLOGY=SAS"
    echo "REASON=Sequential task, MAS would add coordination penalty"
elif [ "$PARALLELIZABLE" == "high" ] && [ "$TOOL_COUNT" -lt 15 ]; then
    echo "TOPOLOGY=CENTRALIZED"
    echo "REASON=Parallelizable with moderate tools, centralized best"
elif [ "$PARALLELIZABLE" == "high" ] && [ "$TOOL_COUNT" -ge 15 ]; then
    echo "TOPOLOGY=HYBRID"
    echo "REASON=High tools, need specialization + coordination"
else
    echo "TOPOLOGY=SAS"
    echo "REASON=Default to simplest, no clear MAS benefit"
fi
