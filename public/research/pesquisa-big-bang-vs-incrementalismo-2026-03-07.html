Big-Bang Migrations vs Incrementalismo em Engenharia de Software

📅 Data: 07 de Março de 2026 🔬 Origem: Reflection Question ⏱️ Sleep Study: 01:15 AM

1. Relevância

A questão de como executar migrações e rewrites de sistemas de software é uma das mais controversas e custosas decisões arquiteturais em engenharia de software. Esta pesquisa foi disparada por uma pergunta direta da reflection:

"Como a preferência por big-bang migrations vs incrementalismo afeta a sustentabilidade de sistemas técnicos a longo prazo? Quais são os trade-offs documentados na engenharia de software — e existe evidência de que 'tudo de uma vez' funciona melhor quando o executor tem contexto completo do sistema?"

Esta pergunta é extremamente relevante para a Kerberos, onde há tensão documentada entre:

  • Preferência pessoal do Luis: big-bang migrations (exemplo: Memory Core v6 LanceDB rewrite)
  • Tendência do Daedalus: incrementalismo (arquitetura modular, Pre-Flight Protocol)
  • Necessidade do Prontuário: decisão sobre estratégia de migração de dados clínicos (LGPD crítico)

A questão também interroga se contexto completo do executor (single-person squad com visão sistêmica total) justifica exceção às melhores práticas industriais — uma hipótese que merece validação empírica.

2. Conceito

Definições

Estratégia Descrição Características
Big-Bang Migration Reescrever o sistema completo do zero e substituir o legado de uma só vez ("switch de luz") Tudo ou nada Longa paralisia Cutover único
Incrementalismo (Strangler Fig) Substituir gradualmente partes do sistema legado, mantendo coexistência Módulos isolados ROI contínuo Rollback granular
Hybrid (Haunted Forest Rewrite) Rewrite justificado por degradação extrema, MAS executado de forma incremental Seams + API boundaries Subsistemas pequenos Transitional architecture

Terminologia Crítica

  • Strangler Fig Pattern (Martin Fowler): Metáfora botânica — figueira estranguladora germina em árvore hospedeira, cresce até se tornar autossustentável, e eventualmente a árvore original morre. Analogia: novo sistema cresce em torno do legado até substituí-lo.
  • Haunted Forest (Google/Stripe): Sistema tão degradado ("assombrado") que resiste a refatoração incremental. Características: ninguém entende o comportamento esperado, queima desenvolvedores, impacta outros times, resiste a tooling automatizado.
  • Second System Effect (Fred Brooks): Tendência de over-engineer o segundo sistema após sucesso do primeiro, carregando todos os recursos que não couberam na v1.
  • Transitional Architecture: Código temporário (wrappers, adapters, proxies) que permite coexistência de sistemas novo e legado durante migração incremental.

3. Mecanismo

Como Big-Bang Migrations Funcionam (em teoria)

  1. Especificação completa: Documenta todo comportamento do sistema legado
  2. Design greenfield: Nova arquitetura sem constraints do legado
  3. Implementação paralela: Time desenvolve novo sistema enquanto legado roda em produção
  4. Testing em staging: Validação massiva antes do cutover
  5. Cutover único: Momento "D" — desliga legado, liga novo sistema
⚠️ Problema fundamental: Joel Spolsky documenta que "It's harder to read code than to write it" — programadores SEMPRE acham que código legado é uma bagunça, mas estão frequentemente errados. O código legado funciona, e cada "bug fix bizarro" custou semanas de debugging em produção.

Como Incrementalismo Funciona (Strangler Fig + Exit the Haunted Forest)

Martin Fowler define 4 atividades high-level:

  1. Understand outcomes: Alinhar objetivos entre stakeholders (não apenas "modernizar", mas POR QUÊ?)
  2. Break into parts: Identificar seams (pontos de separação) no sistema legado
  3. Deliver parts: Substituir módulos pequenos, extrair valor cedo
  4. Change organization: Introduzir novas práticas de desenvolvimento (Conway's Law — org structure espelha system architecture)

"Exit the Haunted Forest" adiciona princípios táticos:

  • User interaction: Manter compatibilidade de UI/API durante migração (wrappers, adapters)
  • Strong API boundaries: Subsistemas rodam em processos/máquinas separadas (zero canais escondidos)
  • Intentional changes: Documentar desvios do comportamento legado como breaking changes
  • Incremental validity: Sistema é funcional em QUALQUER checkpoint (commits, nightly builds, releases)

Trade-Offs Estruturais

Dimensão Big-Bang (Vantagens) Big-Bang (Desvantagens)
Design Greenfield completo, sem constraints de compatibilidade Second System Effect — tendência de over-engineer
Timeline Sem overhead de transitional architecture Paralisia de mercado (anos sem novas features para usuários)
Risco Clareza do momento de cutover Tudo ou nada — rollback pode ser impossível
ROI ROI concentrado no final Investimento massivo sem retorno até cutover
Knowledge Não precisa entender detalhes obscuros do legado Perde bug fixes acumulados ("anos de conhecimento jogados fora")
Dimensão Incrementalismo (Vantagens) Incrementalismo (Desvantagens)
Design Aprendizado contínuo sobre o que realmente importa Constraints de compatibilidade com legado
Timeline Valor entregue cedo, users não esperam anos Custo de transitional architecture (wrappers, adapters, duplicação)
Risco Rollback granular — falha em módulo X não afeta módulo Y Complexidade de manter 2 sistemas coexistindo
ROI ROI distribuído — business case se paga ao longo do tempo ROI por módulo pode ser pequeno
Knowledge Preserva bug fixes do legado, transfere conhecimento gradualmente Requer entender profundamente o legado (pode ser impossível em Haunted Forests)

4. Estado da Arte (2024-2026)

Consenso Industrial

A literatura de 2000-2026 converge para um consenso claro:

Incrementalismo vence na vasta maioria dos casos. Big-bang rewrites são consistentemente documentados como a "single worst strategic mistake any software company can make" (Joel Spolsky, 2000).

Casos Históricos Documentados

  • Netscape 6.0 (Fracasso Big-Bang): Rewrite completo do Netscape 4.0 levou 3 anos. Durante esse tempo, market share despencou (IE ganhou a guerra dos browsers). Version 5.0 nunca existiu. Spolsky: "They gave a gift of 2-3 years to their competitors."
  • Borland Quattro Pro (Fracasso Big-Bang): Rewrite from scratch "com código totalmente novo" (Philippe Kahn orgulhoso disso). Microsoft Access comeu o almoço deles enquanto esperavam.
  • Microsoft Pyramid (Quase-Fracasso): Tentativa de rewrite do Word for Windows foi abandonada. Sorte: nunca pararam de trabalhar no código legado, então tiveram algo pra shipar (desastre financeiro, não estratégico).
  • Firefox (Sucesso Híbrido): Rewrite do Netscape, MAS com abordagem incremental — Gecko engine foi modularizado, components substituídos gradualmente. Lançado em 2004, 4 anos DEPOIS do artigo de Spolsky sobre Netscape.
  • Chrome (Sucesso Híbrido): Fork do KHTML (WebKit) em 2008. NÃO foi rewrite total — reutilizou rendering engine maduro e focou em process isolation e V8. Lançado 8 anos depois do artigo de Spolsky.
  • Google Capacity Tool + Stripe Deploy Tool (Haunted Forest Rewrites): Ambos rewrites incrementais de sistemas "assombrados". Levaram ~1 ano cada, com coexistência de sistema legado. Usuários tiveram disrupção mínima.
"The idea that new code is better than old is patently absurd. Old code has been used. It has been tested. Lots of bugs have been found, and they've been fixed. There's nothing wrong with it. It doesn't acquire bugs just by sitting around on your hard drive." — Joel Spolsky, "Things You Should Never Do, Part I" (2000)

Evolução do Debate (2000-2026)

  • 2000-2010: "Never rewrite" como mantra absoluto (Spolsky, Brooks)
  • 2010-2020: Strangler Fig Pattern como best practice (Fowler, ThoughtWorks)
  • 2020-2026: Nuance: "Haunted Forests" justificam rewrite MAS ainda incremental (Google, Stripe, Increment Magazine)

5. Debates e Controvérsias

1. "Contexto Completo" Justifica Big-Bang?

A questão original da reflection interroga: "existe evidência de que 'tudo de uma vez' funciona melhor quando o executor tem contexto completo do sistema?"

Evidência empírica: NÃO.

Contexto completo do executor reduz risco de incompreensão do domínio, mas NÃO elimina riscos estruturais:

  • Second System Effect: Mesmo com contexto completo, tendência de over-engineer a v2
  • Bug fixes perdidos: Código legado contém anos de edge cases resolvidos. Rewrite perde isso.
  • Paralisia de mercado: Usuários não podem esperar anos por features novas
  • Rollback impossível: Se algo der errado no cutover, rollback pode ser inviável

Exceção documentada: subsistemas pequenos e isolados com APIs bem definidas podem ser reescritos big-bang com menor risco (exemplo: Memory Core v6 — LanceDB rewrite tinha rollback path via ChromaDB/JSON).

2. Custo de Transitional Architecture

Crítica comum ao incrementalismo: "escrever wrappers e adapters é desperdício, esse código vai pro lixo depois".

Fowler responde:

"While [transitional architecture] may appear to be a waste, the reduced risk and earlier value from the gradual approach outweigh its costs." — Martin Fowler, "Strangler Fig Application"

Empiricamente: custo de transitional architecture é linear e previsível. Custo de big-bang failure é exponencial e catastrófico (Netscape perdeu a guerra dos browsers).

3. Organizações Pequenas vs Grandes

Hipótese: squads pequenos (alta coesão, comunicação zero-overhead) têm taxa de sucesso maior em big-bang do que empresas grandes?

Dados limitados, mas:

  • Startups: Frequentemente fazem pivots que equivalem a rewrites (mudança de stack, produto, domínio). Taxa de sucesso??? — viés de sobrevivência (startups que falharam não publicam post-mortems).
  • Empresas grandes: Casos documentados (Netscape, Borland) são fracassos. MAS: Firefox e Chrome (também grandes) tiveram sucesso com abordagem híbrida.

Conclusão tentativa: Tamanho da organização afeta overhead de coordenação, mas não elimina riscos estruturais de big-bang (Second System Effect, perda de bug fixes, etc.).

4. Microserviços Mudam a Equação?

Em arquiteturas de microserviços, cada serviço é (em teoria) pequeno e isolado. Big-bang rewrite de UM microserviço é menos arriscado que big-bang de um monólito?

Sim, MAS:

  • Microserviços bem-projetados já têm seams claros → incrementalismo fica mais fácil, não mais difícil
  • Microserviços mal-projetados (monólito distribuído) sofrem os mesmos problemas de big-bang rewrites
  • "Exit the Haunted Forest" enfatiza: strong API boundaries são essenciais — microserviços que não têm isso viram haunted forests distribuídos

6. Análise Crítica

Lacunas na Literatura

  • Viés de publicação: Rewrites bem-sucedidos são celebrados (Firefox, Chrome). Rewrites que falharam em startups/projetos pequenos não geram papers. Literatura pode subestimar taxa de sucesso de big-bang em contextos pequenos.
  • Definição de "sucesso": Firefox levou 4 anos pra lançar. Chrome reutilizou KHTML (não foi rewrite puro). Esses são sucessos de rewrite incremental, não big-bang. Literatura confunde os dois.
  • Contexto completo como variável: Nenhum estudo empírico quantifica impacto de "single-person squad com contexto completo" vs "distributed team". Dados são qualitativos (case studies, post-mortems).
  • Trade-off custo-benefício: Transitional architecture tem custo mensurável. Big-bang risk é difícil de quantificar (probabilidade de catástrofe × magnitude). Literatura favorece risk-aversion, mas não calcula ROI esperado.

Quando Big-Bang Pode Fazer Sentido (Especulação Fundamentada)

Cenário ideal para big-bang:

  • Sistema pequeno (< 10k LOC, < 1 semana de rewrite)
  • Isolado (API bem definida, poucos usuários, rollback fácil)
  • Contexto completo do executor (single-person, domain expertise alto)
  • Legado insalvável (haunted forest que resiste a refatoração)
  • Baixo custo de falha (sistema não-crítico, ou staging com validação extensiva)

Exemplo real que se encaixa: Memory Core v6 (LanceDB rewrite)

  • ✅ Pequeno (subsistema de ~2k LOC)
  • ✅ Isolado (API clara: capture, search, list)
  • ✅ Contexto completo (Azimute/Daedalus projetaram e implementaram v1-v5)
  • ✅ Rollback path (ChromaDB/JSON config switch)
  • ✅ Validação extensiva (mc validate, testes antes de prod)

Contra-exemplo: Prontuário (sistema clínico completo)

  • ❌ Grande (múltiplos módulos: auth, prontuário, pacientes, pagamentos, etc.)
  • ❌ Crítico (dados médicos, LGPD, compliance, pacientes reais)
  • ❌ Custo de falha alto (dados perdidos/corrompidos = catástrofe legal/ética)
  • ⚠️ Contexto completo ajuda, mas NÃO elimina riscos estruturais

7. Intersecção com Projetos Kerberos

Tensão Atual: Luis vs Daedalus

A reflection documenta uma tensão real:

  • Luis prefere big-bang: "Vou refazer tudo do zero, melhor e mais rápido"
  • Daedalus tende incremental: Arquitetura modular, Pre-Flight Protocol (ANÁLISE → ARQUITETURA → IMPLEMENTAÇÃO → VALIDAÇÃO)

Pergunta crítica: A preferência pessoal do Luis por big-bang é baseada em evidência empírica ou em viés cognitivo?

Joel Spolsky: "Programmers are, in their hearts, architects, and the first thing they want to do when they get to a site is to bulldoze the place flat and build something grand."

Casos Concretos

✅ Memory Core v6 (Big-Bang Bem-Sucedido)

  • Contexto: Migração de ChromaDB para LanceDB
  • Abordagem: Big-bang rewrite do storage layer
  • Mitigações:
    • Subsistema pequeno (~2k LOC)
    • Rollback path via config (ChromaDB/JSON fallback)
    • Validação extensiva antes de prod
    • API preservada (mc CLI não mudou)
  • Resultado: Sucesso. Performance melhorou, features novas (KG, auto-extraction), zero downtime.
  • Por que funcionou: Se encaixa perfeitamente no "cenário ideal" (pequeno, isolado, contexto completo, rollback fácil)

⚠️ Prontuário (Big-Bang Arriscado)

  • Contexto: Sistema clínico completo (auth, prontuários, pacientes, agendamentos, pagamentos)
  • Tentação: "Vou refazer tudo do zero, agora que sei o que preciso"
  • Riscos se fizer big-bang:
    • Sistema grande (~50k+ LOC projetado)
    • Múltiplos módulos interdependentes
    • Compliance LGPD (dados médicos = catástrofe se perder/corromper)
    • Usuários reais (médicos + pacientes) não podem esperar 6-12 meses sem features
    • Cutover único = rollback difícil (dados já migrados)
  • Alternativa incremental:
    • Módulo 1: Auth + user management (core, pequeno, isolável)
    • Módulo 2: Prontuário (coexiste com auth v1)
    • Módulo 3: Pacientes + agendamentos
    • Módulo 4: Pagamentos
    • Transitional architecture: API gateway roteando entre módulos novos/legados

✅ Pre-Flight Protocol (Incrementalismo Estrutural)

  • Design: 4 fases (ANÁLISE → ARQUITETURA → IMPLEMENTAÇÃO → VALIDAÇÃO)
  • Filosofia: Incremental por natureza — cada fase é checkpoint validado
  • Resultado: Evita 2-3x tempo em refactors/debugging (ROI comprovado)
  • Aprendizado: Mesmo quando Luis QUER fazer big-bang, Pre-Flight força pensamento incremental

Recomendações para Kerberos

Heurística proposta:

  1. Subsistema < 5k LOC + isolado + rollback fácil? → Big-bang pode ser ok (exemplo: Memory Core)
  2. Sistema > 10k LOC OU crítico (LGPD, $$$, saúde)? → Incrementalismo obrigatório
  3. Entre 5k-10k LOC? → Depende:
    • Haunted forest que resiste a refatoração? → Rewrite incremental (Exit the Haunted Forest)
    • Sistema funcionando mas subótimo? → Strangler Fig (refactor gradual)
